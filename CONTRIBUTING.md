# 贡献指南

感谢你愿意参与四叶酒馆 / Siye AI / JiuGuanSJ。

这份文档说明如何提交 Issue、贡献代码、改进文档、补充部署经验，以及在改动 AI 角色互动系统时需要注意的边界。

## 适合贡献的方向

| 方向 | 示例 |
| --- | --- |
| 文档 | 修正说明、补充截图、完善 Docker 部署、增加常见问题。 |
| H5 用户端 | 移动端兼容、角色列表体验、聊天交互、个人中心、工单反馈。 |
| 后台管理端 | 表格筛选、角色审核、用户管理、公告、工单、插画管理、权限体验。 |
| 后端服务 | 接口修复、鉴权、角色、会话、消息、权益、订单、模型配置、上传。 |
| AI 集成 | 上下文构建、世界书、记忆刷新、模型供应商配置、SillyTavern 兼容。 |
| 部署 | Docker Compose、Nginx、环境变量模板、日志、备份、服务器迁移。 |
| 安全 | 密钥管理、CORS、上传校验、后台访问控制、支付回调校验。 |

## 提交 Issue

提交问题时请尽量提供：

- 问题发生的模块：`backend`、`admin-web`、`h5-web`、`deploy` 或 `integrations`。
- 复现步骤：从哪个页面或接口开始，执行了什么操作，实际结果是什么。
- 期望结果：你认为正确的行为是什么。
- 环境信息：系统、浏览器、Node.js、Java、Docker、MySQL、Redis 版本。
- 日志或截图：后端日志、浏览器控制台、网络请求、页面截图。

如果问题涉及密钥、真实用户数据、支付信息或私有服务器地址，请不要直接贴到公开 Issue，按 [SECURITY.md](SECURITY.md) 的方式处理。

## 提交 Pull Request

建议流程：

1. Fork 仓库并创建新分支。
2. 保持一次 PR 只解决一个明确问题。
3. 按模块完成开发和自测。
4. 补充必要文档、截图或迁移说明。
5. 在 PR 描述中说明改动动机、影响范围、测试方式和风险点。

PR 描述建议包含：

```text
## 改动内容

## 为什么需要

## 测试方式

## 截图或录屏

## 风险与兼容性
```

## 本地开发命令

后端：

```bash
cd backend
./mvnw spring-boot:run
```

Windows：

```powershell
cd backend
.\mvnw.cmd spring-boot:run
```

后台管理端：

```bash
cd admin-web
npm install
npm run dev
```

H5 用户端：

```bash
cd h5-web
npm install
npm run dev:h5
```

Docker 全栈预览：

```bash
cd deploy
cp .env.example .env
docker compose --env-file .env up -d --build
```

## 代码和文档要求

- 保持改动聚焦，不把无关重构、格式化和功能改动混在一起。
- 新增配置时，同步更新 `.env.example`、`deploy/.env.example` 和相关说明。
- 修改数据库结构时，使用迁移脚本，不直接依赖手工改库。
- 修改后端行为时，尽量补充或更新测试。
- 修改前端可见界面时，提供截图或说明验证过的页面路径。
- 修改 Docker 或 Nginx 时，说明本地验证方式和端口变化。
- 文档优先使用中文，关键配置名、命令、路径保留英文原文。

## 不要提交的内容

不要提交：

- 生产 `.env` 文件。
- 数据库密码、模型供应商密钥、支付密钥、证书、私钥、Token。
- 真实用户数据、聊天记录、支付记录、数据库 dump。
- `node_modules/`、`target/`、`dist/`、`unpackage/`、日志、临时文件、压缩包。
- 未确认授权的图片、音频、Live2D 模型、字体、游戏素材和商业素材。

## 行为边界

不接受用于以下目的的贡献：

- 凭证窃取、绕过鉴权、恶意攻击或持久化后门。
- 欺诈、冒充、骚扰、违法交易或规避监管。
- 未授权采集、泄露或滥用用户数据。
- 明知侵权的素材分发或内容生成流程。

## 开源维护建议

四叶酒馆是一个真实工程项目，后续维护时建议把主线分清楚：

- 私有开发目录可以继续承载作者自己的完整业务迭代。
- 开源目录保持为可公开、可运行、无密钥、无私有素材的发布版本。
- 每次同步开源版本前，先做密钥扫描、素材授权检查、Docker 启动检查和 README 更新。
