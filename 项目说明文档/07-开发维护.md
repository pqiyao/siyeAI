# 开发维护

本文档记录二次开发时最常用的维护约定。

## 分支和提交

建议：

- 主分支保持可运行。
- 功能开发使用独立分支。
- 数据库变更、后端代码、前端页面尽量在同一功能分支内完成。
- 提交前检查不要包含 `.env`、构建产物、上传文件、数据库备份、日志和 IDE 私有配置。

## 后端开发流程

新增后端业务能力通常涉及：

1. 新增或修改 Flyway 迁移。
2. 新增 Entity / DTO / Mapper / XML。
3. 编写 Service 业务逻辑。
4. 编写 Controller API。
5. 如需管理端权限，补充权限目录和后台页面。
6. 增加单元测试或集成测试。

建议优先复用已有包结构，例如角色放在 `character`，聊天放在 `chat`，H5 兼容接口放在 `compat/h5`。

## 数据库变更约定

- 新迁移放到 `backend/src/main/resources/db/migration/`。
- 命名格式：`V序号__英文描述.sql`。
- 已经合并或上线执行过的迁移不要改内容，应新增迁移修正。
- 新增索引时确认查询条件和排序字段。
- 涉及大表回填时避免一次性长事务。

## API 返回和错误处理

项目中有多套接口兼容层：

- 新版 App API 使用 `/api/app/**`。
- H5 兼容 API 使用 `/api/v1/**`。
- 管理后台 API 使用 `/admin/**` 和 `/system/**`。

开发时注意保持对应前端的返回结构，不要随意改变已有字段名称。公共异常处理位于 `shared/error` 和 H5 兼容异常处理类中。

## 前端开发流程

管理端新增页面：

1. 在 `admin-web/src/api/` 添加接口封装。
2. 在 `admin-web/src/views/jiugai/` 添加页面。
3. 按现有 RuoYi 风格接入列表、搜索、表单、弹窗和权限。
4. 如使用后端动态路由或权限目录，同步后端权限配置。

H5 新增页面：

1. 在 `h5-web/pages/` 添加页面。
2. 在 `h5-web/pages.json` 注册路径。
3. 在 `h5-web/common/tavernApi.js` 或对应业务文件中封装 API。
4. 检查浏览器 H5 和 App/Html5Plus 环境差异。

## 聊天和生成能力维护

聊天链路涉及并发控制、SSE、任务状态、消息落库、审计、记忆、世界书和外部模型调用。修改时建议重点回归：

- 普通发送。
- 流式发送。
- 停止生成。
- 继续生成。
- 重新生成。
- swipe 分支切换。
- 图片上传和多模态内容。
- 用户 BYOK 配置。
- 失败日志和错误提示。
- 长期记忆刷新和注入。

## 发布前检查

后端：

```bash
cd backend
./mvnw test
./mvnw package
```

管理端：

```bash
cd admin-web
npm install
npm run build:prod
```

H5：

- 执行 `cd h5-web && npm run build:h5` 构建 H5。
- CLI 构建产物位于 `h5-web/dist/build/h5`；Docker 镜像会自动执行同一构建脚本。

通用检查：

- `git status` 中没有误提交的环境文件或构建产物。
- `.env.example` 和文档中的新增变量保持同步。
- 新增接口已考虑鉴权、限流、CORS 和错误返回。
- 涉及上传、支付、私聊、用户内容的功能已考虑滥用和审核。
- 第三方资源和素材的许可证已确认。

## 开源维护建议

- 保持 README、部署文档和 `.env.example` 同步。
- 如果更换许可证，更新 `LICENSE` 和 `THIRD_PARTY_NOTICES.md`。
- 如果移除或替换第三方资源，在第三方声明中记录。
- 对外发布前不要直接压缩整个工作目录，优先通过 Git 生成干净源码包。
